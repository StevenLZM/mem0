# Mem0 记忆系统高强度学习方案

## 1. 背景与目标

本方案基于 `docs/learning/mem0-memory-system-design.zh-CN.md`，面向已有后端开发、RAG 和 Agent harness engineering 基础的学习者。学习不以运行 Mem0 为终点，而以形成可迁移的记忆系统设计能力为目标。

最终应同时具备三类能力：

1. 独立完成长期记忆系统的需求澄清、分类选型、架构和生产设计；
2. 沿 Mem0 当前源码解释写入、检索、更新、删除、历史和失败边界；
3. 在系统设计面试中清晰表达不变量、取舍、扩展、安全和评测方案。

## 2. 学习约束

- 采用高强度、按掌握度解锁的节奏，不按自然日限速；学习者一天可以完成多个单元。
- 课程预计总投入 25–40 小时，实际速度由过关结果决定。
- 诊断正确率达到 80% 的单元可以压缩讲解，但源码推演、设计取舍和关卡产物不能跳过。
- 采用教练闯关式互动，而非连续灌输式授课。
- 原则上，学习者第一次作答前不提供标准答案；遇到困难时使用分级提示。

## 3. 课程能力架构

### 3.1 关卡一：第一性原理与设计选型

对应教程第 1–3 章，包含四个单元：

1. 记忆系统与 Context、Transcript、Cache、RAG、Profile、Event Log 的边界；
2. 选择、压缩、组织、召回、遗忘五个动作及系统不变量；
3. 按功能、归属、表示、生命周期和检索方式进行多维分类；
4. 根据业务约束选择 Profile、Event Log、RAG、Memory 或组合方案。

过关产物：为 AI 编程助手提交一份简短 ADR，明确记什么、如何表示、归谁、如何召回和何时遗忘。

### 3.2 关卡二：Mem0 架构与完整生命周期

对应教程第 4–8 章，包含五个单元：

5. Monorepo、组件边界和 Provider Factory；
6. 核心数据模型以及 `user_id`、`agent_id`、`run_id`、`actor_id` 的职责；
7. 从消息到抽取、去重、Embedding、主存、History 和 Entity 的写入生命周期；
8. semantic threshold、BM25、entity boost 和 reranker 的检索顺序；
9. ADD-only、显式更新、删除、过期、历史和审计冲突。

过关产物：不看教程画出完整生命周期，提交失败矩阵，并定位关键源码符号。

### 3.3 关卡三：从零实现 V0–V4

对应教程第 9 章、`examples/memory_system_design_demo.py` 和配套测试，包含三个单元：

10. 从保存全部消息演进到原子事实、scope 和语义检索；
11. 加入关键词、实体和融合评分，并识别伪多路召回；
12. 加入历史、更新、删除、过期和冲突表示。

过关产物：改造教学引擎，并用测试证明 scope、去重、召回、过期和历史行为。

### 3.4 关卡四：生产化与评测

对应教程第 10–12 章，包含三个单元：

13. 幂等、重试、补偿、对账和背压；
14. 容量估算、分片、索引迁移、多租户隔离、隐私和遗忘权；
15. 写入质量、Recall、MRR、NDCG、时间一致性、任务收益、成本和延迟。

过关产物：完成生产设计评审，包括故障矩阵、SLO、容量估算和离线/在线评测方案。

### 3.5 关卡五：综合答辩

对应教程第 13–14 章，包含一个单元：

16. 在需求不完整、故障追问和百万用户约束下完成端到端系统设计。

过关产物：完整系统设计方案、源码答辩和面试复盘。

## 4. 单课互动流程

每个单元按以下顺序闭环：

1. **课前诊断**：先回答 2–4 个问题，确定可压缩和需要深入的部分；
2. **核心讲解**：按直觉、模型、备选、选型、数据流、源码和取舍展开；
3. **源码定位**：先预测行为，再核对文件、类、函数和失败路径；
4. **即时练习**：至少包含概念辨析、源码或故障推演、业务设计三类题；
5. **反馈追问**：指出正确推理、遗漏的不变量和危险假设；
6. **过关检查**：用自己的语言总结，并提交决策表、数据流图、伪代码、测试或设计说明。

答错时采用三级提示：

1. 指出问题属于哪个设计维度；
2. 给出反例或失败场景；
3. 引导回到相关源码或教程段落。

## 5. 评分与进度

每个单元按四个维度评分，每项 0–3 分：

- 概念模型：是否理解设计原因；
- 源码证据：是否能以当前 Mem0 实现支持结论；
- 设计取舍：是否能比较替代方案及失败影响；
- 迁移能力：是否能应用到新的业务场景。

总分至少 9/12 且任何一项不得为 0，才算过关。安全、scope、授权和删除边界出现致命错误时，必须补课后重答。

每个单元结束后输出进度卡，记录：单元状态、四维评分、已掌握内容、暴露的误区和下一步任务。

## 6. 动手实验策略

- 阅读、问答、设计题和源码推演直接在当前对话中进行。
- 需要修改代码时使用独立学习分支，避免实验污染 `main`。
- 实验优先使用确定性教学引擎，不依赖网络或真实 API Key。
- DeepSeek、Qdrant 和真实 Provider 作为后期可选实验，不作为基础过关条件。
- 代码实验必须测试通过，并能说明测试证明了哪个不变量、没有证明什么。

## 7. 最终完成标准

课程以一次 45–60 分钟的系统设计答辩结束。学习者需要独立完成：

1. 澄清需求、约束和非目标；
2. 定义 scope、证据、权限和删除不变量；
3. 设计事件、事实、历史、实体和索引模型；
4. 说明写入、检索、更新和删除路径；
5. 分析部分失败、一致性、补偿和对账；
6. 完成容量、成本和延迟估算；
7. 设计多租户安全、隐私和被遗忘权流程；
8. 给出组件评测、端到端指标和上线策略；
9. 回答 Mem0 当前源码的边界追问。

最终评分至少达到 75%，且不得存在跨租户访问、错误删除、密钥泄露或把 best-effort 误称为事务/自动最终一致等致命问题。

## 8. 非目标

- 不以背诵 Mem0 API、Provider 清单或公式为学习成果；
- 不要求从头通读整个 monorepo；
- 不把教学 demo 当作生产实现；
- 不以真实外部服务可用性作为课程是否完成的判断标准；
- 不在尚未理解设计约束前追求复杂图存储、自动时间推理或分布式部署。
