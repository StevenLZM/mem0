# Mem0 记忆系统学习总结

> 基于 [Mem0 记忆系统高强度学习方案](docs/superpowers/specs/2026-07-21-mem0-intensive-learning-design.md)。
>
> 本文只记录各单元的核心模型、设计边界、源码证据与关键取舍，用于快速复习。

---

## 单元 1：记忆载体与真相源

### 一句话模型

> Context 服务于当前推理；Long-term Memory 保存面向未来任务的派生事实和经验，但通常不是权威真相源。

### 状态载体的边界

| 载体 | 核心职责 | 典型生命周期 |
| --- | --- | --- |
| **Context** | 本次模型推理的输入 | 单次请求或短会话 |
| **Transcript** | 保留原始对话证据 | 按会话或合规期限保留 |
| **Cache** | 保存可重建的计算结果 | TTL 或 LRU |
| **RAG** | 检索版本化的外部文档知识 | 跟随文档版本 |
| **Profile** | 保存经确认的结构化当前状态 | 属性有效期 |
| **Event Log** | 保存不可变事件，支持审计和重放 | 长期或法规期限 |
| **Long-term Memory** | 保存面向未来复用的派生事实和经验 | TTL、长期、衰减或显式删除 |

### 核心设计原则

1. **持久化不等于真相源。** 真相源由业务所有权和冲突裁决权决定，不由数据是否长期保存决定。
2. **Long-term Memory 通常是派生状态。** 它不应取代 Profile、配置库、Event Log 或正式文档。
3. **冲突必须有确定性策略。** 权威优先级、失效、更新、删除和对账缺一不可。

### 本地入口与远程客户端

| 类 | 执行边界 |
| --- | --- |
| `Memory` / `AsyncMemory` | 在应用进程中编排 LLM、Embedder、Vector Store 等组件 |
| `MemoryClient` / `AsyncMemoryClient` | 通过 HTTP 调用远程 Mem0 服务 |

> 本地编排不代表所有 Provider 和数据都在本机：OpenAI、Qdrant 等具体 Provider 仍可以是远程服务。

### 派生记忆与推理

```text
原始证据 → 推理或抽取 → 筛选与验证 → 派生记忆
                       └→ 未验证猜测留在当前任务
```

- **推理**服务于当前问题，可以包含尚未验证的猜测和中间假设。
- **派生记忆**只应保存经过筛选、对未来有复用价值的事实、决策和经验，并保留 scope、来源、时间和纠错边界。
- 不应将模型的完整思维过程作为长期记忆。

### Profile 与派生记忆的冲突处理

```text
Profile 权威状态变更
        ├→ 读取时权威覆盖
        ├→ 异步更新或失效派生记忆
        └→ 定期对账与修复
```

当 Profile 与 Long-term Memory 冲突时，应以 Profile 中经用户确认的当前状态为准。低风险偏好不一定需要跨存储强事务，但读取优先级只能作为安全网，不能掩盖长期陈旧数据。

> 涉及隐私删除时，仅在读取时忽略数据远远不够；必须实际删除所有派生副本。

## 单元 2：五个动作与系统不变量

### 一句话模型

> 记忆系统不是“把文本写入向量库”，而是在 scope、来源、时间、权限和删除约束下，持续执行选择、压缩、组织、召回和遗忘。

### 记忆系统的五个动作

| 动作 | 需要回答的问题 | 主要风险 |
| --- | --- | --- |
| **选择** | 什么值得记？ | 把临时猜测当成稳定事实，或遗漏关键决策 |
| **压缩** | 用事件、摘要、原子事实还是步骤表示？ | 压缩误差、语境丢失和无法纠错 |
| **组织** | 归谁、何时发生、如何索引？ | scope 错误、时间混用和跨租户泄漏 |
| **召回** | 当前任务允许且需要哪些记忆？ | 未授权召回、排序错误和陈旧事实污染 |
| **遗忘** | 何时过期、更新、降权或删除？ | 副本残留、审计断裂和隐私违规 |

```text
消息或事件 → 选择 → 压缩 → 组织与索引 → 按作用域召回 → 注入 Context
                                  └→ 过期、更新、降权或删除
```

### 六个系统不变量

1. **Scope 和授权过滤先于相似度。** 先确定允许访问的候选集，再在其中排序。
2. **原始证据与派生记忆可区分。** 派生事实需要能追溯、纠错和重建。
3. **创建、更新、事件和过期时间不混用。** 同一记忆可以同时具有这些不同时间语义。
4. **部分失败可观察、可重试、可修复。** 不能把跨多存储的 best-effort 误报为事务成功。
5. **删除覆盖全部派生副本。** 主存、向量索引、实体关联和缓存不得留下可召回内容。
6. **召回结果仍是不可信输入。** 它不能绕过权限检查，也不能被直接当作可执行指令。

### Scope 的当前源码边界

- [`_validate_and_trim_entity_id()`](mem0/memory/main.py#L148) 将非字符串 ID 转为字符串，去除首尾空白，并拒绝空值、纯空白和内部空白。
- [`_build_filters_and_metadata()`](mem0/memory/main.py#L287) 将所有已提供的 `user_id`、`agent_id` 和 `run_id` 同时加入写入 metadata 和查询 filters。
- 三种 scope ID 全部缺失时抛出 `Mem0ValidationError`，错误码为 `VALIDATION_001`。
- `user_id` 是机器标识符，不是显示名称；项目应使用 `project_id` metadata 或应用层作用域，不应冒充 `agent_id`。
- Scope 过滤实现候选集限定，但不能取代 API 鉴权和项目成员权限检查。

### 短期状态、当前事实与历史证据

| 内容 | 建议处理 |
| --- | --- |
| “今晚临时试一下 Rust” | 放入 Run 级工作记忆，并设置过期或在任务结束时清理 |
| “Atlas 当前统一使用 TypeScript” | 作为经授权项目成员可召回的当前事实 |
| “Atlas 以前使用 Python 3.12” | 标记为已被取代；只在历史或审计语义下召回，不参与默认当前事实召回 |
| API Key 等密钥 | 在进入 Transcript、日志和记忆抽取链路前脱敏，真实值交给 Secret Manager |

```text
项目规范：Python 3.12 → TypeScript
             │              └→ 当前可召回事实
             └→ 已被取代的历史证据
```

Alice 和 Bob 是否能召回 Atlas 的新项目规范，由项目 scope 与成员授权共同决定，不由语义相似度决定。

### 删除与部分失败

如果主数据库已删除，但向量索引或实体关联仍能召回原内容，就不能宣称删除完成。系统应使用 tombstone 立即阻断读取，记录失败副本，执行幂等重试，并通过告警和对账修复残留。

## 单元 3：功能分类与工程检索

### 一句话模型

> 记忆类型由未来如何使用决定，而不是由消息的表面形式决定；同一段经历可以分别派生工作、语义、情景和程序记忆。

### 快速判断方法

- 有具体时间和事件经过：情景记忆。
- 去掉具体事件后仍然成立的事实：语义记忆。
- 描述应该怎样执行：程序记忆。
- 只服务当前任务：工作记忆。

### 四种功能类型

| 类型 | 回答的问题 | 常见 Scope | 典型生命周期 | 主要检索方式 |
| --- | --- | --- | --- | --- |
| **工作记忆** | 当前正在做什么？ | Run/Task | 当前任务结束、取消或超时 | 直接读取、最近状态 |
| **语义记忆** | 稳定事实是什么？ | User、Project、Organization | 长期，但允许更新和取代 | 精确字段、关键词、向量 |
| **情景记忆** | 何时发生过什么？ | Project、Organization、User | 复用或审计期限 | 时间、实体、关键词、向量 |
| **程序记忆** | 应该怎样做？ | Agent、Project、Organization | 流程版本有效期 | 适用条件、版本、关键词、向量 |

Scope 回答“这是谁的记忆，谁有权使用”，不能与其他字段混淆：

- `event_type` 是分类或检索字段，不是 Scope。
- `version` 是生命周期和适用条件，不是 Scope。
- `project_id`、`user_id`、`agent_id` 和 `run_id` 表达不同的所有权与使用边界。
- 权限回答“能否访问”，优先级回答“多个允许使用的值冲突时以谁为准”，二者不能混用。

### 一个事件可以派生多种记忆

```text
Qdrant 维度故障
    ├→ 情景记忆：何时发生、触发条件、症状、处理和结果
    ├→ 语义记忆：集合维度必须与 Embedding 输出维度一致
    └→ 程序记忆：更换模型前检查维度，并迁移或重建集合
```

这些记录不是简单重复，因为它们回答不同问题，也需要不同的 Scope、生命周期和检索策略。

查询“Atlas 上次发生 Qdrant 写入失败是怎么回事”时，应以情景记忆为主：

1. 根据“上次”找到最近一次相关事故；
2. 读取事故发生时间、症状、触发条件、处理和结果；
3. 补充事故发生时有效的语义事实，不能用当前配置改写过去；
4. 如果用户还问“怎么修”，再补充程序记忆。

### 语义记忆的结构化表示

语义记忆适合表示成“主体—属性—值”，并保存状态、有效期和来源：

| 字段 | 示例 |
| --- | --- |
| Scope | `project:atlas` |
| Subject | `project:atlas` |
| Predicate | `backend_language` |
| Value | `TypeScript` |
| Status | `active` |
| Valid time | `valid_from` / `valid_to` |
| Source | 配置、ADR 或原始消息引用 |
| Authority | 已确认事实或派生判断 |

当 TypeScript 被 Go 取代时，旧记录应标记为 `superseded` 并结束有效区间，新记录成为 `active`。结构化主存负责事实、来源和状态，向量、BM25 和实体索引只是可重建的检索投影。

### 情景记忆与索引

情景记录通常包含：

- 事件时间和事件类型；
- 涉及的项目、组件和其他实体；
- 症状、错误码和触发条件；
- 处理动作、结果和根因状态；
- 原始日志、事故单或部署事件引用。

不同查询需要不同索引：

| 查询意图 | 主要索引 |
| --- | --- |
| 最近一次 Atlas 故障 | Project＋事件时间结构化索引 |
| 精确错误码 | 精确字段或 BM25/倒排索引 |
| “换模型后写不进去” | 向量索引 |
| 所有与 Qdrant 相关的事故 | 实体索引 |

Event Log 保存原始不可变事件；情景记忆是经过整理的可检索经历。未确认根因必须标记为假设，不能冒充确定事实。

### LLM 与 Memory Service 的职责

```text
LLM
├→ 理解用户意图
├→ 选择逻辑查询方法
└→ 输出受约束的参数

Memory Service
├→ 从认证上下文注入可信 Scope
├→ 执行权限和 Schema 校验
├→ 查询结构化主存与检索索引
├→ 去重、融合和排序
└→ 返回带来源的有限上下文
```

LLM 不应直接操作数据库或自行决定可信的 `tenant_id`、内部 `project_id` 和访问权限。它可以根据 Tool Schema 输出逻辑查询计划，由 Memory Service 翻译成 SQL、BM25、向量或实体查询。

写入时，LLM 可以提出候选事实、实体和摘要；Memory Service 负责校验、去重、冲突处理、状态变更和最终持久化。未验证候选即使暂存，也不能在变为 `active` 前参与正常召回。

### 同一集合与差异化检索

四类记忆可以共享物理向量基础设施，但不能采用统一的全局 Top-K：

- 不同类型具有不同的 Scope、权限、TTL、Schema 和更新语义；
- 工作记忆强调当前状态，语义记忆强调有效事实，情景记忆强调时间，程序记忆强调版本和适用条件；
- 单一类型数量过多时，可能占满 Top-K，挤掉其他关键记忆；
- 不同类型的相似度分数分布也不一定可以直接比较。

推荐先按类型路由和分别召回，再使用候选配额、分数归一化、去重和融合排序。物理上可以共用集合，逻辑隔离和差异化策略不能省略。

### 删除与跨 Scope 数据

删除个人记忆时，应删除或匿名化用户的私人对话、个人偏好及其派生引用。独立的项目事故和操作流程可以继续保留，但其中不必要的姓名、邮箱或私人原文必须清理。

> 不能仅通过把记录标成 Project scope，就绕过个人数据删除要求。
